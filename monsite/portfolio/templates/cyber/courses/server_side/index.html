🧪 Web Fuzzing

🎯 Définition

Les server side attacks ciblent l'application ou le service fourni par un serveur, tandis que les attaques coté client se produisent sur la machine du client et non sur le serveurlui meme. Comprendre et identifi ces differences sera etentile en pentest et bubounty Par exemples les vulnérabilité comme le Cross-SiteScriptiong (XSS) ciblent le navigateur web c'est a dire le client à l'inverse les attaques coré serveur ciblent le serveur web.

🎯 Les différents types de failles côté serveur

  • Server-Side Request Forgery (SSRF)
  • Server-Side Template Injection (SSTI)
  • Server-Side Includes (SSI) Injection
  • eXtensible Stylesheet Language Transformations (XSLT) Server-Side Injection

Server-Side Request Forgery (SSRF)

La Server-Side Request Forgery (SSRF) est une vulnérabilité qui permet à un attaquant de manipuler une application web afin de lui faire envoyer des requêtes non autorisées depuis le serveur lui‑même. Cette faille apparaît généralement lorsqu’une application effectue des requêtes HTTP vers d’autres serveurs en utilisant des données fournies par l’utilisateur.

Lorsqu’elle est exploitée avec succès, une SSRF peut permettre à un attaquant d’accéder à des systèmes internes, de contourner des pare-feux ou encore de récupérer des informations sensibles.

Server-Side Template Injection (SSTI)

Les applications web utilisent souvent des moteurs de templates et des modèles côté serveur pour générer dynamiquement des réponses, comme du contenu HTML. Cette génération repose fréquemment sur des données fournies par l’utilisateur, permettant à l’application de s’adapter dynamiquement à ses actions.

Lorsqu’un attaquant parvient à injecter du code dans un template, une vulnérabilité de type Server-Side Template Injection (SSTI) peut se produire. Une SSTI peut entraîner divers risques de sécurité, notamment la fuite de données et, dans les cas les plus graves, la compromission complète du serveur via l’exécution de code à distance.

XSLT Server-Side Injection

La XSLT Server-Side Injection est une vulnérabilité qui apparaît lorsqu’un attaquant parvient à manipuler les transformations XSLT exécutées sur le serveur. Le langage XSLT (Extensible Stylesheet Language Transformations) est utilisé pour transformer des documents XML en d’autres formats, comme du HTML, et il est fréquemment employé dans les applications web pour générer du contenu dynamiquement.

Dans le cadre d’une injection XSLT côté serveur, l’attaquant exploite des faiblesses dans la manière dont les transformations XSLT sont traitées. Cela peut lui permettre d’injecter et d’exécuter du code arbitraire sur le serveur, entraînant des risques majeurs tels que la fuite de données ou la compromission complète du système.

Introduction à la SSRF

Les vulnérabilités de type Server-Side Request Forgery (SSRF) font partie du Top 10 de l’OWASP. Ce type de vulnérabilité apparaît lorsqu’une application web récupère des ressources distantes en se basant sur des données fournies par l’utilisateur, comme une URL.

Server-Side Request Forgery

Lorsqu’un serveur web récupère des ressources distantes en fonction d’une entrée utilisateur, un attaquant peut potentiellement forcer le serveur à effectuer des requêtes vers des URL arbitraires qu’il contrôle. Dans ce cas, le serveur est vulnérable à une SSRF.

Même si cela peut sembler anodin au premier abord, selon la configuration de l’application web, une SSRF peut avoir des conséquences dévastatrices, comme nous le verrons dans les sections suivantes.

De plus, si l’application web repose sur un schéma ou un protocole d’URL fourni par l’utilisateur, un attaquant peut provoquer des comportements encore plus dangereux en manipulant ce schéma. Voici quelques exemples de schémas d’URL couramment utilisés dans l’exploitation des vulnérabilités SSRF :

  • http:// et https:// : Ces schémas permettent de récupérer du contenu via des requêtes HTTP/S. Un attaquant peut les utiliser pour contourner des WAF, accéder à des endpoints restreints ou atteindre des services internes.
  • file:// : Ce schéma permet de lire un fichier présent sur le système local. Un attaquant peut l’exploiter pour lire des fichiers locaux sur le serveur (LFI).
  • gopher:// : Ce protocole permet d’envoyer des octets arbitraires à une adresse donnée. Un attaquant peut l’utiliser pour envoyer des requêtes HTTP POST avec un payload personnalisé ou communiquer avec d’autres services, comme des serveurs SMTP ou des bases de données.

Pour plus de détails sur les techniques avancées d’exploitation SSRF, comme les contournements de filtres ou le DNS rebinding, consultez le module Modern Web Exploitation Techniques.

📂 Directory Fuzzing

Permet de découvrir des dossiers cachés comme : /admin, /backup, /dev

Commande ffuf

ffuf -w /usr/share/seclists/Discovery/Web-Content/directory-list-2.3-medium.txt \
-u http://TARGET/FUZZ

🔎 Le mot-clé FUZZ est remplacé par chaque entrée de la wordlist.

📄 File Fuzzing

Permet de trouver des fichiers sensibles : .php, .txt, .bak, .js 

ffuf -w common.txt \
-u http://TARGET/FUZZ \
-e .php,.html,.txt,.bak,.js

⚠️ Exemple :
config.php.bak → fuite de credentials

📚 Wordlists (très important)

Les wordlists sont la base du fuzzing.

  • common.txt → rapide
  • directory-list-2.3-medium.txt → standard
  • big.txt → très complet
/usr/share/seclists/Discovery/Web-Content/

🧠 Méthodologie (mon workflow)

  1. Recon → analyser le site
  2. Directory fuzzing
  3. File fuzzing
  4. Analyse des réponses
  5. Pivot vers exploitation

💡 Toujours regarder :

  • Status code (200, 403, 500)
  • Taille réponse
  • Différences comportementales

🌳 Recursive Fuzzing

Jusqu’ici, on a fuzz uniquement un niveau (root ou dossier). Mais les applications web peuvent avoir une structure profonde avec plusieurs dossiers imbriqués.

Le recursive fuzzing permet d’automatiser l’exploration complète.

⚙️ Fonctionnement

  1. Fuzz du répertoire principal
  2. Détection des dossiers valides
  3. Relance automatique du fuzz dans chaque dossier

📌 Exemple : 

ffuf -w directory-list-2.3-medium.txt \
-u http://TARGET/FUZZ \
-e .html \
-recursion

🧠 Avantages :

  • Gain de temps énorme
  • Découverte de structures profondes
  • Moins d'effort manuel

⚠️ Options utiles : 

-recursion-depth 2
-rate 500
-timeout 10

🧩 Parameter & Value Fuzzing

Ici on ne fuzz plus des dossiers mais des paramètres.

GET parameters

http://site.com/page?x=FUZZ

Exemple avec wenum : 

wenum -w common.txt \
-u "http://TARGET/get.php?x=FUZZ" \
--hc 404

POST parameters

ffuf -u http://TARGET/post.php \
-X POST \
-H "Content-Type: application/x-www-form-urlencoded" \
-d "y=FUZZ" \
-w common.txt \
-mc 200

🎯 Objectif :

  • Trouver des valeurs valides
  • Contourner des contrôles
  • Déclencher des comportements cachés

🌐 VHost & Subdomain Fuzzing

Permet de découvrir des sites cachés sur un même serveur.

VHost (Gobuster)

gobuster vhost \
-u http://target \
-w common.txt \
--append-domain

Subdomain

gobuster dns \
-d example.com \
-w subdomains.txt

🎯 Exemple :

  • admin.example.com
  • dev.example.com
  • test.example.com

🎯 Filtering (très important)

Le fuzzing génère beaucoup de bruit. Il faut filtrer pour voir l’essentiel.

FFUF

-mc 200        # match code
-fc 404        # exclude code
-fs 0-1000     # filter size
-mt >500       # réponse lente

Exemple : 

ffuf -u http://TARGET/FUZZ \
-w wordlist.txt \
-mc 200 \
-fc 404

💡 Sans filtre = trop de faux positifs

✅ Validation des résultats

Tous les résultats ne sont pas des failles.

Il faut valider manuellement.

Étapes

  1. Reproduire la requête
  2. Analyser la réponse
  3. Tester l’impact
curl http://TARGET/backup/

🔎 Vérifier :

  • Content-Length
  • Content-Type
  • Présence de fichiers sensibles

⚠️ Toujours rester éthique (CTF / lab uniquement)

🔌 Web APIs

Une API permet à des applications de communiquer entre elles.

Types

  • REST → JSON, endpoints multiples
  • SOAP → XML, structure stricte
  • GraphQL → requêtes flexibles

Exemple REST : 

GET /users/123

🧠 Différence avec un site classique :

  • Site → HTML
  • API → JSON / XML

🎯 Identification des endpoints

Avant de fuzz une API, il faut trouver ses endpoints.

Méthodes

  • Documentation (Swagger)
  • Analyse réseau (Burp Suite)
  • Fuzzing des paramètres

Exemple : 

/users
/users/123
/products/456

🧩 SOAP & GraphQL Fuzzing

🔎 Découverte des endpoints SOAP

Les API SOAP utilisent des fichiers WSDL pour définir leur structure.

  • Opérations disponibles (endpoints)
  • Paramètres d’entrée et de sortie
  • Types de données
  • URL du service

🛠️ Méthodes :

  • Analyse du fichier WSDL
  • Sniff réseau (Wireshark, tcpdump)
  • Fuzzing des paramètres XML

⚡ GraphQL

Contrairement à REST, GraphQL utilise un seul endpoint (souvent /graphql).

📥 Exemple de requête

query {
  user(id: 123) {
    name
    email
  }
}

✏️ Mutation

mutation {
  createPost(title: "Test") {
    id
  }
}

🔍 Découverte :

  • Introspection GraphQL
  • Documentation (GraphiQL)
  • Analyse du trafic réseau

🚀 API Fuzzing

Le API fuzzing consiste à tester automatiquement les endpoints d’une API avec des requêtes modifiées pour détecter des vulnérabilités.

🎯 Objectifs

  • Trouver des endpoints cachés
  • Tester la robustesse
  • Détecter des injections
  • Simuler un attaquant réel

🧪 Types de fuzzing API

1️⃣ Parameter Fuzzing

Tester différentes valeurs pour un paramètre :

?id=FUZZ

2️⃣ Data Format Fuzzing

Modifier le JSON / XML :

{
  "id": "AAAAA"
}

3️⃣ Sequence Fuzzing

Tester l’ordre des requêtes (important pour les APIs complexes).

🔍 Découverte d’API (Endpoints)

Une API peut contenir des endpoints cachés non documentés.

📚 Exemple d’API

  • GET / → accueil
  • GET /items/{id}
  • POST /items
  • PUT /items/{id}
  • DELETE /items/{id}

⚠️ Certains endpoints peuvent être cachés !

🛠️ Fuzzer une API

git clone https://github.com/PandaSt0rm/webfuzz_api.git
cd webfuzz_api
pip3 install -r requirements.txt

python3 api_fuzzer.py http://IP:PORT

📊 Résultat

Found valid endpoints:
- /docs
- /secret_endpoint

💡 Endpoint caché trouvé → accès possible à une flag

✅ Validation des résultats

Le fuzzing génère souvent des faux positifs. Il faut toujours vérifier manuellement.

🛠️ Méthode

  1. Reproduire avec curl
  2. Analyser la réponse
  3. Confirmer la vulnérabilité

📌 Exemple

curl http://IP:PORT/backup/

Si tu vois une liste de fichiers → vulnérabilité confirmée ⚠️

🔎 Headers utiles

curl -I http://IP:PORT/backup/password.txt
  • Content-Type
  • Content-Length

💡 Permet d’analyser sans exposer les données sensibles
