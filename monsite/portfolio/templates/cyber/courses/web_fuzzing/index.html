🧪 Web Fuzzing

🎯 Définition

Le web fuzzing est une technique qui consiste à envoyer des entrées inattendues ou aléatoires à une application web afin de détecter des comportements anormaux et des vulnérabilités.

⚔️ Fuzzing vs Brute Force

Fuzzing : large spectre, teste des entrées variées et inattendues.

Brute force : approche ciblée, teste toutes les possibilités d’une valeur.

Fuzzing → tester plein d'entrées différentes
Brute force → tester toutes les combinaisons d'un mot de passe

🧠 Analogie :
- Fuzzing = jeter tout ce que tu trouves sur une porte 🚪 - Brute force = essayer toutes les clés 🔑

🔥 Pourquoi faire du fuzzing ?

  • Découvrir des endpoints cachés
  • Identifier des failles (XSS, SQLi, IDOR…)
  • Automatiser les tests
  • Simuler un attaquant réel
  • Améliorer la sécurité globale

🧠 Concepts essentiels

Concept Description Exemple
Wordlist Liste de mots utilisée pour fuzz admin, login, backup
Payload Donnée envoyée ' OR 1=1 --
Fuzzer Outil automatisé ffuf, gobuster
Response Réponse du serveur 200, 403, 500

🛠️ Outils principaux

  • ffuf → rapide et flexible
  • Gobuster → simple et efficace
  • Feroxbuster → scan récursif
  • wfuzz / wenum → fuzzing avancé de paramètres

Installation rapide

sudo apt update
sudo apt install golang python3 python3-pip pipx

go install github.com/ffuf/ffuf/v2@latest
go install github.com/OJ/gobuster/v3@latest

📂 Directory Fuzzing

Permet de découvrir des dossiers cachés comme : /admin, /backup, /dev

Commande ffuf

ffuf -w /usr/share/seclists/Discovery/Web-Content/directory-list-2.3-medium.txt \
-u http://TARGET/FUZZ

🔎 Le mot-clé FUZZ est remplacé par chaque entrée de la wordlist.

📄 File Fuzzing

Permet de trouver des fichiers sensibles : .php, .txt, .bak, .js 

ffuf -w common.txt \
-u http://TARGET/FUZZ \
-e .php,.html,.txt,.bak,.js

⚠️ Exemple :
config.php.bak → fuite de credentials

📚 Wordlists (très important)

Les wordlists sont la base du fuzzing.

  • common.txt → rapide
  • directory-list-2.3-medium.txt → standard
  • big.txt → très complet
/usr/share/seclists/Discovery/Web-Content/

🧠 Méthodologie (mon workflow)

  1. Recon → analyser le site
  2. Directory fuzzing
  3. File fuzzing
  4. Analyse des réponses
  5. Pivot vers exploitation

💡 Toujours regarder :

  • Status code (200, 403, 500)
  • Taille réponse
  • Différences comportementales

🌳 Recursive Fuzzing

Jusqu’ici, on a fuzz uniquement un niveau (root ou dossier). Mais les applications web peuvent avoir une structure profonde avec plusieurs dossiers imbriqués.

Le recursive fuzzing permet d’automatiser l’exploration complète.

⚙️ Fonctionnement

  1. Fuzz du répertoire principal
  2. Détection des dossiers valides
  3. Relance automatique du fuzz dans chaque dossier

📌 Exemple : 

ffuf -w directory-list-2.3-medium.txt \
-u http://TARGET/FUZZ \
-e .html \
-recursion

🧠 Avantages :

  • Gain de temps énorme
  • Découverte de structures profondes
  • Moins d'effort manuel

⚠️ Options utiles : 

-recursion-depth 2
-rate 500
-timeout 10

🧩 Parameter & Value Fuzzing

Ici on ne fuzz plus des dossiers mais des paramètres.

GET parameters

http://site.com/page?x=FUZZ

Exemple avec wenum : 

wenum -w common.txt \
-u "http://TARGET/get.php?x=FUZZ" \
--hc 404

POST parameters

ffuf -u http://TARGET/post.php \
-X POST \
-H "Content-Type: application/x-www-form-urlencoded" \
-d "y=FUZZ" \
-w common.txt \
-mc 200

🎯 Objectif :

  • Trouver des valeurs valides
  • Contourner des contrôles
  • Déclencher des comportements cachés

🌐 VHost & Subdomain Fuzzing

Permet de découvrir des sites cachés sur un même serveur.

VHost (Gobuster)

gobuster vhost \
-u http://target \
-w common.txt \
--append-domain

Subdomain

gobuster dns \
-d example.com \
-w subdomains.txt

🎯 Exemple :

  • admin.example.com
  • dev.example.com
  • test.example.com

🎯 Filtering (très important)

Le fuzzing génère beaucoup de bruit. Il faut filtrer pour voir l’essentiel.

FFUF

-mc 200        # match code
-fc 404        # exclude code
-fs 0-1000     # filter size
-mt >500       # réponse lente

Exemple : 

ffuf -u http://TARGET/FUZZ \
-w wordlist.txt \
-mc 200 \
-fc 404

💡 Sans filtre = trop de faux positifs

✅ Validation des résultats

Tous les résultats ne sont pas des failles.

Il faut valider manuellement.

Étapes

  1. Reproduire la requête
  2. Analyser la réponse
  3. Tester l’impact
curl http://TARGET/backup/

🔎 Vérifier :

  • Content-Length
  • Content-Type
  • Présence de fichiers sensibles

⚠️ Toujours rester éthique (CTF / lab uniquement)

🔌 Web APIs

Une API permet à des applications de communiquer entre elles.

Types

  • REST → JSON, endpoints multiples
  • SOAP → XML, structure stricte
  • GraphQL → requêtes flexibles

Exemple REST : 

GET /users/123

🧠 Différence avec un site classique :

  • Site → HTML
  • API → JSON / XML

🎯 Identification des endpoints

Avant de fuzz une API, il faut trouver ses endpoints.

Méthodes

  • Documentation (Swagger)
  • Analyse réseau (Burp Suite)
  • Fuzzing des paramètres

Exemple : 

/users
/users/123
/products/456

🧩 SOAP & GraphQL Fuzzing

🔎 Découverte des endpoints SOAP

Les API SOAP utilisent des fichiers WSDL pour définir leur structure.

  • Opérations disponibles (endpoints)
  • Paramètres d’entrée et de sortie
  • Types de données
  • URL du service

🛠️ Méthodes :

  • Analyse du fichier WSDL
  • Sniff réseau (Wireshark, tcpdump)
  • Fuzzing des paramètres XML

⚡ GraphQL

Contrairement à REST, GraphQL utilise un seul endpoint (souvent /graphql).

📥 Exemple de requête

query {
  user(id: 123) {
    name
    email
  }
}

✏️ Mutation

mutation {
  createPost(title: "Test") {
    id
  }
}

🔍 Découverte :

  • Introspection GraphQL
  • Documentation (GraphiQL)
  • Analyse du trafic réseau

🚀 API Fuzzing

Le API fuzzing consiste à tester automatiquement les endpoints d’une API avec des requêtes modifiées pour détecter des vulnérabilités.

🎯 Objectifs

  • Trouver des endpoints cachés
  • Tester la robustesse
  • Détecter des injections
  • Simuler un attaquant réel

🧪 Types de fuzzing API

1️⃣ Parameter Fuzzing

Tester différentes valeurs pour un paramètre :

?id=FUZZ

2️⃣ Data Format Fuzzing

Modifier le JSON / XML :

{
  "id": "AAAAA"
}

3️⃣ Sequence Fuzzing

Tester l’ordre des requêtes (important pour les APIs complexes).

🔍 Découverte d’API (Endpoints)

Une API peut contenir des endpoints cachés non documentés.

📚 Exemple d’API

  • GET / → accueil
  • GET /items/{id}
  • POST /items
  • PUT /items/{id}
  • DELETE /items/{id}

⚠️ Certains endpoints peuvent être cachés !

🛠️ Fuzzer une API

git clone https://github.com/PandaSt0rm/webfuzz_api.git
cd webfuzz_api
pip3 install -r requirements.txt

python3 api_fuzzer.py http://IP:PORT

📊 Résultat

Found valid endpoints:
- /docs
- /secret_endpoint

💡 Endpoint caché trouvé → accès possible à une flag

✅ Validation des résultats

Le fuzzing génère souvent des faux positifs. Il faut toujours vérifier manuellement.

🛠️ Méthode

  1. Reproduire avec curl
  2. Analyser la réponse
  3. Confirmer la vulnérabilité

📌 Exemple

curl http://IP:PORT/backup/

Si tu vois une liste de fichiers → vulnérabilité confirmée ⚠️

🔎 Headers utiles

curl -I http://IP:PORT/backup/password.txt
  • Content-Type
  • Content-Length

💡 Permet d’analyser sans exposer les données sensibles
